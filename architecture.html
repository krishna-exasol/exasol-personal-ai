One command to run an Exasol Personal database with an MCP access layer and JSON-native SQL.

Install — one command  ·  macOS (Apple Silicon)
$ curl -fsSL https://raw.githubusercontent.com/krishna-exasol/exasol-personal-ai/main/install.sh | sh
Requires Docker Desktop running. First run installs the exasol launcher and the local database (10–20 min).

This bundle brings together three official Exasol components behind a single install.sh. The twist: Exasol Personal is a launcher, not a Docker image — so the database runs on the host, and only the two companion tools are containers that connect to it.

Key idea
The database is not a service in compose.yaml. The installer asks the launcher where the database is (exasol info --json), writes that into .env, and the two containers connect outward to the host via host.docker.internal.

Components

ComponentWhat it isRuns asWhy
Exasol Personal Go launcher CLI that provisions a real Exasol DB (local VM or cloud) Host process + managed VM Not a container image; it manages its own DB lifecycle. We consume the endpoint it produces.
MCP Server exasol-mcp-server pip package, HTTP mode on :4896 Container Clean pip install; needs only client→DB connectivity, which works fine to a host DB.
JSON Tables Python CLI that shells out to a Rust (cargo) ingest engine Container (standing; CLI exec'd) Needs the repo + Rust toolchain at runtime; isolated due to a conflicting pyexasol pin.
Why two containers, not one
MCP needs pyexasol>=1,<2; JSON Tables needs pyexasol>=2.2,<3. Those can't coexist in one Python environment — and the tools don't talk to each other, only to the database — so each runs isolated.

Why Personal changes the design

Nano bundle (previous)

Exasol Nano is a Docker image. All three components were containers on one Docker network; the DB was just another service, reachable by name.

Personal bundle (this)

Exasol Personal is a launcher. Its local database is macOS-Apple-Silicon-only and lives in a managed VM. There is nothing to put in compose — so the DB stays on the host and the containers reach in.

Install flow

  1. Check prerequisites — Docker engine running; warn if the host isn't macOS/arm64.
  2. Ensure the launcher — install exasol from downloads.exasol.com if it isn't on PATH or in ~/.local/bin.
  3. Ensure a database — if none exists, run exasol install local (10–20 min first time); if stopped, exasol start.
  4. Discover the connection — exasol info --json → host/port/user; secrets.json → password.
  5. Stage files & write .env into ~/.exasol-personal-ai/.
  6. Build & start containers — docker compose up -d --build; generate the run-json-tables.sh helper.

Connection discovery

The local database port is dynamic, so nothing is hardcoded. The installer reads:

# host/port/user from the launcher
exasol info --json
  → connection.host, connection.dbPort, connection.username

# password from the deployment secrets
~/.exasol/personal/deployments/default/secrets.json
  → dbPassword

…and writes them to ~/.exasol-personal-ai/.env, which compose.yaml consumes:

EXA_DB_HOST=host.docker.internal
EXA_DB_PORT=<discovered>
EXA_USER=sys
EXA_PASSWORD=<discovered>

TLS certificate validation is disabled for both tools because the local DB uses a self-signed certificate. Redeploying the DB changes the port — re-running install.sh re-discovers it and rewrites .env.

The ingest caveat

Exasol's bulk import uses HTTP transport: the CLI opens a local endpoint and the database connects back to it to pull data. With the DB on the host (in Personal's VM) and JSON Tables in a container, that reverse path isn't guaranteed — so ingest may fail even though queries succeed.

Host-mode fallback: run the JSON Tables CLI directly on the Mac host, where it shares 127.0.0.1 with the database:

git clone https://github.com/exasol-labs/exasol-json-tables.git
cd exasol-json-tables && python3 -m pip install -e .   # needs Python 3.10+ & Rust
exasol-json-tables ingest-and-wrap --input data.json --name my_events \
  --dsn 127.0.0.1:<dbPort> --user sys --password <password>

Platform & the cloud path

Personal's install local is macOS Apple Silicon only (enforced by the launcher). The launcher itself runs cross-platform for cloud targets, so on other systems you can provision a remote DB and point the bundle at it:

exasol install aws        # or azure | exoscale | stackit
exasol info               # note host + dbPort

EXASOL_SKIP_DB_DEPLOY=1 EXASOL_DB_HOST=<host> EXASOL_DB_PORT=<port> ./install.sh

Security defaults

  • MCP runs read-only (writes disabled in mcp-settings.json) and is published only on 127.0.0.1:4896.
  • --no-auth keeps the local MCP endpoint open for local LLM clients — do not expose the port beyond localhost.
  • Credentials live only in ~/.exasol-personal-ai/.env; keep the directory private.
